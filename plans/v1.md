# YOMI Hustle LLM Arena - Implementation Plan v1

This plan supersedes the remaining open questions in `plans/v0.md` and translates the two reviews in `code-review/` into a cutover-oriented execution plan for coding agents.

It assumes the v0 foundation has landed: daemon protocol/models, baseline adapters, artifact writing, Godot bridge scaffolding, observation/legal-action extraction, validation/fallback plumbing, single-match orchestration, and tournament basics. The focus of v1 is to close the gap between "well-tested components" and "provable live end-to-end play in the real game."

## Planning assumptions

- The repository now has a credible MVP foundation, but live game play is not yet proven end to end.
- The highest-priority failures are integration failures, not missing scaffolding: the live wire contract, native action-application path, match lifecycle hooks, and in-engine verification.
- v1 is a protocol and implementation cutover. Backwards compatibility with the current live contract is not required; daemon and mod may move together to a new versioned surface.
- All bridge-affecting work must be proven in the real Godot/game environment, not only through Python mirror harnesses.
- Strategic completeness is in scope for v1: richer payload semantics, prediction support, observation history, missing fighter state, stronger provider traces, and full local operator workflow.
- Alien-specific gameplay support is intentionally out of scope for this plan. Other currently supported characters remain in scope.
- The live workflow target is one supported developer/operator path that is scripted and reproducible; full cross-platform installer automation is not required for v1.

## Global implementation rules

- Protocol changes must remain schema-first and versioned under `schemas/`.
- v1 work should introduce a new protocol/schema cutover version rather than preserving legacy message forms.
- Match artifacts must continue to be written under `runs/<timestamp>_<match_id>/` with enough metadata to audit every decision and every fallback.
- Bridge messages after handshake must always use the canonical envelope shape. Bare payloads are not an accepted runtime contract in v1.
- Canonical serialization for `state_hash` and `legal_actions_hash` must be shared across Python and GDScript well enough to produce identical hashes for identical logical payloads.
- Mod-side legality remains authoritative, but daemon-side validation and prompt contracts must be upgraded to fully describe supported payloads and extras.
- Any change that relies on decompiled game internals must add either in-engine proof or explicit runtime compatibility checks.
- Real-provider smoke coverage should be opt-in and disabled by default in normal local test runs.
- Provider credentials must come from environment variables or ignored local config files; committed config examples must never contain secrets.
- Python dependency management and execution should continue to use `uv`.

## Recommended execution order

- Start with `WU-015` and `WU-016` in parallel.
- Land `WU-017` immediately after the protocol cutover and apply-path proof are in place.
- Once `WU-015` and `WU-017` land, split gameplay-surface work across `WU-018` and `WU-019`.
- Once the new protocol, payload semantics, and observation model are stable, land `WU-020`.
- `WU-021` should be treated as the first "claimable live MVP" gate and should not start until `WU-015` through `WU-020` are complete.
- `WU-022`, `WU-023`, and `WU-024` are late-stage quality, operator-safety, and throughput work and can proceed in parallel once `WU-021` lands.

## WU-015 - Protocol v2 cutover and wire-contract enforcement

- Blocked by: none
- Goal: replace the current partially drifted live bridge contract with a single versioned envelope-only protocol that both the mod and daemon enforce consistently.

### Scope

- Introduce a new versioned schema surface for envelope, decision request, action decision, event, match ended, and any related payloads touched by the cutover.
- Update the daemon and mod to speak only the new versioned contract for live traffic after handshake.
- Remove acceptance of bare `decision_request` and `action_decision` payloads from the live path.
- Add protocol fields needed by later v1 work, including explicit support for richer payload specs and prediction-capable extras.
- Define and document canonical serialization rules used for hashing in both Python and GDScript.

### Implementation details known ahead of time

- The mod must envelope turn requests in the same way it already envelopes handshake traffic.
- Events and `match_ended` should stay on the same canonical envelope contract as turn requests and decisions.
- This work unit should make the cutover explicit in docs and schemas rather than leaving transitional ambiguity.
- Legacy compatibility shims should be removed from live code paths once the new contract is in place.

### Acceptance criteria

- [x] A live mod can complete handshake and send enveloped turn requests that the daemon accepts without protocol warnings.
- [x] The daemon sends only enveloped action decisions, and the mod validates only the new contract in the live path.
- [x] Canonical hash serialization is specified once and implemented consistently in both languages.
- [x] Protocol documentation clearly states that the previous bare-payload behavior is obsolete.

### Tests to add

- [x] Add schema tests covering valid and invalid examples of every protocol message touched by the cutover.
- [x] Add daemon integration tests proving enveloped turn requests succeed and bare payloads are rejected.
- [x] Add mod harness or in-engine tests proving outbound turn requests, events, and `match_ended` use the canonical envelope.
- [x] Add cross-language hash fixtures that assert identical hashes for identical canonical payloads.

### Execution notes for future agents

- `v2` is now the only supported live protocol version; `v1` schema files remain on disk for historical reference and regression coverage.
- The canonical hash implementation is shared by fixture in `tests/fixtures/protocol/canonical_hash_cases.json`, with daemon assertions in `tests/daemon/test_protocol_validation.py` and mod-harness assertions in `tests/mod/test_observation_harness.py`.
- The live mod path now requires enveloped `action_decision` responses; bare decision payloads are intentionally rejected in both the GDScript validator mirror and the daemon integration suite.
- `ProtocolCodec.gd` is the mod-side source of truth for envelope building and canonical JSON hashing; later gameplay work should reuse it instead of ad hoc `to_json()` hashing.

## WU-016 - Native action pipeline proof and apply-path hardening

- Blocked by: none
- Goal: prove, in engine, whether direct `queued_*` mutation is sufficient for real move commitment, or replace it with the native action-selection path if it is not.

### Scope

- Trace the real action-selection and lock-in flow around `action_selected`, ready-state changes, queued fields, and any downstream commit signals.
- Add targeted in-engine instrumentation or smoke scenes that can compare UI/native-AI action commitment with bridge-driven action commitment.
- If direct queued-field mutation is incomplete, update `ActionApplier.gd` to participate in the native path instead of bypassing it.
- Document the final authoritative action-application contract in `docs/game_internals.md`.

### Implementation details known ahead of time

- This work unit is intentionally narrow: prove the current path or replace it. It should not expand into broader gameplay feature work.
- Evidence from the decompile output should be treated as guidance, but the final acceptance gate is observed in-engine behavior.
- Ready-state, turn locking, and simultaneous-decision side effects are part of scope because they determine whether the game actually accepts the injected choice.

### Acceptance criteria

- [x] There is in-engine proof that bridge-applied actions commit through the same effective lifecycle as native selections.
- [x] If additional native hooks are required, `ActionApplier.gd` uses them and no longer relies on incomplete field mutation.
- [x] The documented action-application path is specific enough for later agents to extend safely.

### Tests to add

- [x] Add an in-engine smoke test that applies at least one simple action through the bridge and verifies the game advances through the expected ready/commit path.
- [x] Add an in-engine comparison test or fixture proving that bridge-applied and native-applied actions produce the same relevant queued/ready outcomes.
- [x] Add regression coverage for simultaneous-turn situations where both players become actionable together.

### Execution notes for future agents

- `ActionApplier.gd` now treats `on_action_selected(action, data, extra)` as the authoritative live apply hook recovered in `tests/fixtures/decompile/yomi_hustle_supported_build_16151810.json`; direct `queued_*` writes remain only as a compatibility fallback.
- The headless proof path is `godot3 --no-window --path mod --script res://BridgeActionApplySmoke.gd`, with pytest coverage in `tests/mod/test_action_apply_smoke.py`.
- The smoke script uses lightweight native-hook doubles to assert parity between direct native application and bridge-driven application, plus a two-player ready coordinator to catch simultaneous-turn regressions.
- Godot 3.5 requires `Reference`, not `RefCounted`, for bridge helper scripts loaded by the smoke harness.

## WU-017 - Match lifecycle hooks, runtime compatibility checks, and identifier hygiene

- Blocked by: `WU-015`, `WU-016`
- Goal: make the live mod lifecycle explicit and safe by emitting authoritative match-end data, verifying supported game internals at runtime, and removing brittle session identifiers.

- Add runtime compatibility checks for the supported game build, scene paths, signals, and critical fighter/state properties before enabling the bridge.
- Surface compatibility failures clearly in logs and mod state rather than failing silently.
- Replace the current ad hoc UUID generation with a compliant and testable identifier helper.

### Implementation details known ahead of time

- The bridge should disable itself safely when required runtime hooks are unavailable instead of attempting partial live control.
- Compatibility checks should focus on the minimum set of paths and properties that the bridge actually depends on.
- Match-end emission must integrate with the artifact/finalization path already expected by the daemon.

### Acceptance criteria

- [x] A live match emits `match_ended` with authoritative winner and end-reason data instead of finalizing as a disconnect in the happy path.
- [x] Unsupported or drifted game builds are detected before live turn control begins.
- [x] Match and session identifiers are generated through a deterministic, testable helper instead of the current biased implementation.

### Tests to add

- [x] Add in-engine or harness tests for successful `match_ended` emission and final artifact finalization.
- [x] Add negative tests for missing required scene nodes, missing signals, or unsupported game-version markers.
- [x] Add unit tests for the new identifier helper.

### Execution notes for future agents

- The supported live match-resolution seam for public build `16151810` is `res://game.gd`: `game_ended()` fires before `game_won(winner)`, and `end_game()` computes winner plus `game_end_tick`. The repository fixture now records that under `tests/fixtures/decompile/yomi_hustle_supported_build_16151810.json`.
- Mod-side runtime gating lives in `mod/YomiLLMBridge/bridge/RuntimeCompatibility.gd`. The current minimum contract is `Global.VERSION == "1.9.20-steam"`, engine version prefix `3.5.1`, `player_actionable`/`game_ended`/`game_won`, both action-button nodes, and the fighter `queued_*` plus `on_action_selected(...)` surface.
- `mod/YomiLLMBridge/bridge/TurnHook.gd` is now responsible for both compatibility gating and authoritative `match_ended` emission. It snapshots `user://replay/autosave` at match start and reports the newest newly-written `.replay` file when one appears.
- `mod/YomiLLMBridge/bridge/Telemetry.gd` now sends a dedicated `match_ended` envelope after emitting the audit event, so the daemon finalization path sees the canonical end-of-match payload instead of an event-only approximation.
- Daemon-side identifier generation now lives in `daemon/src/yomi_daemon/ids.py`. `SessionIdGenerator` owns live connection IDs, and `new_match_id()` / `new_benchmark_id()` centralize prefixed token formatting for future callers.
- The mod-side Python mirror for this work unit is `scripts/mod_runtime_harness.py`; harness coverage is in `tests/mod/test_runtime_harness.py`, and daemon helper coverage is in `tests/daemon/test_ids.py`.

## WU-018 - Structured move parameterization, prediction support, and legality validation

- Blocked by: `WU-015`
- Goal: upgrade action payloads from UI-widget labels to structured legal-action contracts that models, baselines, and validators can use to fully play parameterized moves.

### Scope

- Replace the current `payload_spec` shape with structured constraints including field names, types, ranges, defaults, enum choices, requiredness, and semantic hints where recoverable.
- Extend legal actions and action-decision extras to support prediction in addition to DI, feint, and reverse.
- Update the mod and daemon validators to validate payload data against the structured spec instead of only checking for "dict present."
- Upgrade baseline payload synthesis and legality-aware fallback logic to operate on the richer contract.
- Cover character-specific parameterized actions already in scope for Cowboy, Robot, Ninja, Mutant, and Wizard.

### Implementation details known ahead of time

- The schema surface should be expressive enough for slider, option, checkbox, directional, and XY-style payloads without embedding engine widgets into the protocol.
- Semantic descriptions should help prompt rendering but should not become a substitute for concrete value constraints.
- This work unit should make unsupported payload fields fail loudly and predictably rather than silently passing through to the game.

### Acceptance criteria

- [x] Legal actions expose structured payload constraints detailed enough for a model or baseline to build valid payloads for supported parameterized moves.
- [x] Prediction is represented consistently in schemas, protocol models, prompts, validators, and application code.
- [x] Invalid payload keys, types, ranges, and unsupported extras are rejected before they reach the game.
- [x] Parameterized actions no longer degrade routinely into fallback because the contract is underspecified.

### Tests to add

- [x] Add schema and typed-model tests for structured payload specs and prediction-capable extras.
- [x] Add mod and daemon validation tests for invalid keys, invalid ranges, invalid enum values, and unsupported extras.
- [x] Add baseline adapter tests for payload synthesis across slider, enum, checkbox, directional, and XY-style parameter shapes.
- [x] Add fixture-driven tests for character-specific parameterized moves for every in-scope character except Alien.

### Execution notes for future agents

- `payload_spec` now uses a structured descriptor contract rooted at `{"type":"object","additionalProperties":false,"properties":...}` rather than the old widget-tag map. Schema validation for that contract lives in `schemas/decision-request.v2.json`, and daemon-side request-relative enforcement lives in `daemon/src/yomi_daemon/validation.py`.
- Prediction support is no longer just a boolean toggle. `LegalAction` carries optional `prediction_spec`, `ActionDecision.extra.prediction` uses the shared structured object contract, and the default spec source is `default_prediction_spec()` in `daemon/src/yomi_daemon/protocol.py`.
- Baseline and fallback payload synthesis now resolve structured defaults for slider, enum, checkbox, directional, and XY/object payloads. The daemon path is `daemon/src/yomi_daemon/adapters/baseline.py`; the mod mirror is `scripts/mod_decision_harness.py` and `mod/YomiLLMBridge/bridge/FallbackHandler.gd`.
- Character fixture coverage for parameterized actions now exists under `tests/fixtures/game_state/` for Cowboy, Robot, Ninja, Mutant, and Wizard. Those fixtures drive `tests/mod/test_observation_harness.py` and are the current source of truth for supported structured payload shapes in tests.
- The repository `uv` project is rooted in `daemon/`, not the repo root. The verified quality-gate commands for this work unit were `uv run ruff format` and `uv run ruff check` from the repo root, then `uv sync`, `PYTHONPATH=/Users/nlevine/Dev/yomi-ai uv run ty check src ../tests ../scripts`, and `uv run pytest ../tests` from `daemon/`.

## WU-019 - Observation v2 completeness and canonical state extraction

- Blocked by: `WU-015`, `WU-017`
- Goal: give policies a strategically meaningful and technically correct observation surface, and ensure the same logical state hashes identically across languages.

### Scope

- Replace placeholder or incorrect observation fields, including the current `history: []`, `hitstun`, `hitlag`, and object typing logic.
- Add a bounded sliding history window containing recent turn-level actions, results, and any cheap tactical outcomes needed for prediction-heavy play.
- Add missing high-value fighter state where available: air movements remaining, feints remaining, initiative, grounded state, sadness or penalty indicators, wakeup throw immunity, combo proration, and any similar universal fields recovered from the game.
- Expand character-resource extraction for the currently supported cast except Alien, while keeping the schema additive within the new v2 observation contract.
- Implement a shared canonical serializer for state and legal-action hashing across Python and GDScript.

### Implementation details known ahead of time

- History should stay compact and deterministic; it is intended to improve policy context, not become a full replay log.
- Object identification should use game-meaningful identifiers rather than base engine class names wherever practical.
- Any field whose meaning is uncertain should be documented explicitly rather than mislabeled.

### Acceptance criteria

- [x] Observation fields previously mislabeled or crash-prone are corrected or removed.
- [x] The observation includes bounded recent history and the missing universal fighter state needed for better tactical decisions.
- [x] Object typing is gameplay-meaningful enough for policies to distinguish major object classes.
- [x] Cross-language canonical serialization produces identical hashes for equivalent observation and legal-action payloads.

### Tests to add

- [x] Add fixture-driven tests for each new or corrected observation field.
- [x] Add regression tests proving `hitstun` and `hitlag` read from the correct source or are omitted when unavailable.
- [x] Add cross-language fixture tests for canonical serialization and hash identity.
- [x] Add object-identification tests covering at least projectile-like and install-like objects present in fixtures.

### Execution notes for future agents

- `hitstun` was renamed to `blockstun` throughout — it was always reading `blockstun_ticks` from the engine, so the old name was misleading. The schema, protocol models, GDScript builder, Python harness, and all tests were updated.
- `grounded` is derived from `position.y <= 0.0` since the engine has no explicit grounded flag; ObservationBuilder.gd uses `fighter.position.y <= 0.0`.
- Object type resolution chain in GDScript: strip trailing digits from node name → script filename → `get_class()`. The Python harness mirrors this with `classify_object_type()` using `OBJECT_CATEGORY_MAP`.
- Objects with `"unknown"` category omit the `category` key entirely to keep payloads clean.
- Optional advanced fighter fields (`air_actions_remaining`, `feints_remaining`, `initiative`, `sadness`, `wakeup_throw_immune`, `combo_proration`) are only emitted when present in game state, guarded by `"field_name" in fighter` checks in GDScript.
- `HistoryEntry` dataclass added to protocol.py; `Observation.history` changed from `tuple[JsonObject, ...]` to `tuple[HistoryEntry, ...]`. MAX_HISTORY_ENTRIES = 10.
- New fixture `tests/fixtures/game_state/observation_v2_rich.json` exercises all new fields including airborne fighter, multiple object types, and character-specific data.
- Canonical hash fixture expanded from 2 to 4 cases in `tests/fixtures/protocol/canonical_hash_cases.json`.
- Schema `schemas/decision-request.v2.json` now requires `type` and `position` on objects (breaking for old payloads missing these).
- The validation test object fixture needed updating from `{"id": ..., "kind": ...}` to `{"type": ..., "category": ..., "position": ...}` format.
- Quality gate: all 332 tests pass, ruff format/check clean, ty check clean.

### Acceptance criteria

- [x] Prompt contracts match the live protocol and no longer under-specify DI or supported extras.
- [x] The real daemon server path persists prompt traces and provider payload traces when logging is enabled.
- [x] Missing credentials, provider failures, and malformed provider output still degrade cleanly into structured fallbacks.
- [x] A documented opt-in smoke path exists for at least one real provider.

### Tests to add

- [x] Add prompt rendering tests covering DI ranges, prediction support, and structured payload-spec rendering.
- [x] Add server-path integration tests proving prompt traces are written during real orchestration, not only through adapter helpers.
- [x] Add opt-in smoke coverage for a real provider using live credentials supplied through the environment.
- [x] Add tests for token or cost metadata capture when providers expose it.

### Execution notes for future agents

- The DI range was already correct at `[-100, 100]` in `decision_output_json_schema()` in `daemon/src/yomi_daemon/prompt.py`; the `[-1, 1]` mismatch noted in the plan had been fixed in an earlier work unit.
- The critical server-path fix: `server.py:_handle_decision_request()` now calls `resolve_adapter_decision()` (from `orchestrator.py`) instead of calling `resolve_policy_decision()` directly with `adapter.decide`. This ensures prompt traces and provider payloads are persisted to `prompts.jsonl` during live server operation when `logging.prompts` is enabled.
- `resolve_adapter_decision` handles both decision persistence and prompt trace persistence in one call, so the server no longer calls `writer.append_decision()` separately.
- Fallback detection in the server is now derived from `decision.fallback_reason is not None` rather than from the `OrchestratedDecision.used_fallback` flag, since `resolve_adapter_decision` returns `PolicyDecisionResult` not `OrchestratedDecision`.
- The opt-in smoke test is in `tests/daemon/test_real_provider_smoke.py`. Run with `YOMI_SMOKE_PROVIDER=anthropic ANTHROPIC_API_KEY=sk-... uv run pytest tests/daemon/test_real_provider_smoke.py -v`. Supports `anthropic`, `openai`, and `openrouter` providers.
- Token metadata tests are in `tests/daemon/test_token_metadata.py`, covering both Anthropic and OpenAI adapter token capture, artifact persistence through `resolve_adapter_decision`, and baseline adapter null-token behavior.
- Quality gate: all 344 tests pass, ruff format/check clean, ty check clean.

## Parallelization notes

- `WU-015` and `WU-016` are intentionally independent so protocol cutover and apply-path proof can proceed at the same time.
- After `WU-015`, payload-contract work (`WU-018`) can proceed while lifecycle and compatibility work (`WU-017`) lands.
- `WU-019` should wait for `WU-017` because runtime compatibility and lifecycle hooks affect what observation fields are safe to trust live.
- `WU-020` should wait for the new protocol, payload, and observation surfaces so prompt changes do not need to be reworked twice.
- `WU-021` is the first end-to-end proof gate and should stay serialized behind the earlier contract work.
- `WU-022`, `WU-023`, and `WU-024` can branch after live end-to-end proof exists.
