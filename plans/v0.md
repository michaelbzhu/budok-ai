# YOMI Hustle LLM Arena - Implementation Plan v0

This plan translates `specs/unified_spec.md` into a commit-oriented execution plan for coding agents.

## Planning assumptions

- The repository currently contains only specifications, so early work units must establish project scaffolding, tooling, and test harnesses.
- The mod and daemon should evolve in parallel, but anything that depends on exact YOMI Hustle hook points must wait for game mapping output.
- Every work unit is intended to be a logical standalone commit with clear deliverables, tests, and dependency boundaries.
- The game remains the authority on legality; the daemon remains the authority on policy orchestration, logging, and tournament management.
- MVP transport is localhost WebSocket with the mod acting as client and the daemon acting as server.
- CI is intentionally deferred until after MVP stability so implementation can optimize for fast local iteration and low process overhead.

## Global implementation rules

- Protocol changes must remain schema-first and versioned under `schemas/`.
- Match artifacts must be written under `runs/<timestamp>_<match_id>/` with enough metadata to audit every decision.
- All mod-applied decisions must be revalidated against live legal actions before they are injected.
- All Python dependency management and Python code execution should use `uv` (for example `uv sync`, `uv run pytest`, `uv run python ...`) rather than raw `pip`, `python`, or `pytest` invocations.
- Tests should be added as close as possible to the code they protect, with reusable fixtures under `tests/fixtures/`.
- Work units should prefer additive changes so multiple agents can work concurrently with minimal merge conflicts.
- After completing a work unit and before committing it, update that work unit with an `Execution notes for future agents` section capturing non-obvious implementation facts that arose during execution and are not already in the plan.

## Recommended execution order

- Start with `WU-000`, `WU-001`, and `WU-002`.
- Once `WU-000` and `WU-002` land, daemon work can split across `WU-003` to `WU-007`.
- Once `WU-000` and `WU-001` land, mod work can split across `WU-008` to `WU-010`.
- End-to-end runner, tournaments, and hardening should wait for both daemon and mod MVP paths to exist.

## WU-002 - Canonical protocol schemas and typed models

- Blocked by: `WU-000`
- Goal: define the shared wire contract once so daemon and mod work can proceed against a stable versioned interface.

### Scope

- Create versioned JSON schemas for envelope, hello/hello-ack, decision request, action decision, event, match ended, and config payloads.
- Implement typed Python protocol models and validation helpers in `daemon/src/yomi_daemon/protocol.py` and `validation.py`.
- Define canonical enums and field constraints for message type, protocol version, decision type, fallback reason, and bounded DI vectors.
- Document schema semantics in `docs/protocol.md`, including handshake compatibility rules and append-only versioning policy.

### Implementation details known ahead of time

- Envelope shape should be `{type, version, ts, payload}`.
- `DecisionRequest` must include `match_id`, `turn_id`, `player_id`, `deadline_ms`, `state_hash`, `legal_actions_hash`, `decision_type`, `observation`, and `legal_actions`.
- `ActionDecision` must include `action`, `data`, and `extra`, with optional debug metadata.
- Hashes should be treated as opaque strings in the schema layer; computation logic belongs elsewhere.

### Acceptance criteria

- [x] Every required protocol message in the spec has a concrete schema file and a typed Python representation.
- [x] Validation helpers reject malformed payloads with actionable error messages.
- [x] Protocol documentation explains compatibility, negotiation, and the distinction between schema validation and mod-side legality validation.

### Tests to add

- [x] Add unit tests for valid and invalid examples of every core message type.
- [x] Add round-trip tests from Python typed model -> JSON payload -> schema validation.
- [x] Add negative tests for stale or mismatched version fields and malformed `extra.di` vectors.

### Execution notes for future agents

- The canonical v1 protocol surface now lives in `daemon/src/yomi_daemon/protocol.py` and `daemon/src/yomi_daemon/validation.py`; later work units should extend those models before introducing ad hoc payload dicts.
- `ProtocolModel.to_dict()` omits unset optional fields by default and only serializes explicit JSON `null` for contract-critical nullable fields such as `ActionDecision.data`, `DecisionExtras.di`, and `MatchEnded.winner`.
- `scripts/check_schemas.py` now runs `Draft202012Validator.check_schema(...)` in addition to JSON parsing, so broken schema structure will fail the existing schema smoke test immediately.
- The v1 protocol currently defines a single `decision_type` enum member, `turn_action`; add new decision kinds as append-only enum extensions unless a breaking protocol version is being introduced.

- [x] Unsupported protocol or schema versions are rejected cleanly without crashing the server.

### Tests to add

- [x] Add async integration tests for successful handshake, unsupported version rejection, and localhost binding defaults.
- [x] Add tests for player-slot policy mapping in `HelloAck`.

### Execution notes for future agents

- `daemon/src/yomi_daemon/match.py` now carries the session skeleton that later routing and artifact work should extend: accepted protocol version, daemon version, player-slot mapping, optional config snapshot, and game/mod/schema metadata.

- [x] Baselines return structured `ActionDecision` payloads that validate against schemas.
- [x] Deterministic baselines produce identical outputs for the same seeded request sequence.

### Tests to add

- [x] Add unit tests per baseline policy covering empty metadata, tie-breaking, and unsupported extras.
- [x] Add determinism tests across repeated seeded runs.
- [x] Add contract tests that every adapter returns decisions matching the typed protocol model.

### Execution notes for future agents

- Adapter construction now lives in `daemon/src/yomi_daemon/adapters/base.py`; `build_policy_registry(...)` materializes configured adapters once per runtime config, and `build_player_policy_adapters(...)` maps those instances onto `PlayerSlot.P1` / `PlayerSlot.P2`.
- Baseline implementations live in `daemon/src/yomi_daemon/adapters/baseline.py` and currently support `baseline/random`, `baseline/block_always`, `baseline/greedy_damage`, and `baseline/scripted_safe`.
- Deterministic seeding is derived from `policy_id`, `trace_seed`, `match_id`, `player_id`, `turn_id`, `state_hash`, and `legal_actions_hash` using SHA-256; this keeps baseline decisions stable across Python processes without relying on interpreter hash randomization.
- Baselines prefer legal actions whose `payload_spec` can be concretely synthesized from simple descriptor shapes (`const`, `default`, `enum`/`choices`, primitive `type`, `properties`, `items`) and otherwise fall back to `data: null`.
- Conservative extras are synthesized uniformly today: `extra.di` becomes `(0, 0)` only when the chosen action supports DI, while `feint` and `reverse` stay `false` unless a later work unit teaches a policy to opt into them.

## WU-007 - Response parsing, daemon-side validation, and fallback engine

- Blocked by: `WU-002`, `WU-004`, `WU-006`
- Goal: centralize how the daemon parses model outputs, validates them against the request, and constructs explicit fallback decisions.

### Scope

- Implement `response_parser.py`, `fallback.py`, and validation utilities for matching `ActionDecision` values to the current request.
- Support provider-native structured output, JSON parsing, and constrained text extraction in descending preference order.
- Implement a single retry-with-correction path for invalid LLM output when configured.
- Encode layered fallback behavior: `safe_continue`, optional `heuristic_guard`, then `last_valid_replayable` if enabled and legal.

### Implementation details known ahead of time

- Daemon-side validation is advisory; mod-side legality checks remain final.
- Fallback outputs must carry explicit reason codes for timeout, disconnect, malformed output, illegal output, or stale response.
- `last_valid_replayable` should only reuse prior decisions when the same action remains legal under the current request.

### Acceptance criteria

- [x] The daemon can turn invalid or late model output into a structured fallback decision without throwing uncaught exceptions.
- [x] Retry behavior is bounded and config-driven.
- [x] Validation detects mismatched `match_id`, `turn_id`, unsupported extras, and actions absent from the legal set.

### Tests to add

- [x] Add unit tests for structured parse success, JSON extraction, text extraction, correction retry, and fallback selection.
- [x] Add negative tests for stale IDs, illegal actions, malformed payloads, and out-of-bounds DI vectors.
- [x] Add property-style tests for `last_valid_replayable` legality checks.

### Execution notes for future agents

- `daemon/src/yomi_daemon/response_parser.py` now accepts provider-native structured objects, wrapped JSON strings, and constrained key/value text; it normalizes missing `match_id`, `turn_id`, `data`, and default `extra` fields from the live `DecisionRequest` before schema and legality validation.
- Bounded correction retry is configured through `PolicyConfig.options["response_parser"]` with `enable_correction_retry` and `max_correction_retries`; the current implementation intentionally caps retries at `0` or `1` to match the spec's single correction pass.
- Request-relative decision legality now lives in `daemon/src/yomi_daemon/validation.py`; `validate_action_decision_for_request(...)` enforces stale ID rejection, legal action membership, supported extras, and conservative payload-shape checks, while `is_replayable_decision(...)` reuses the same checks without requiring matching request IDs.
- `daemon/src/yomi_daemon/fallback.py` treats fallback modes as an ordered capability set: `safe_continue` chooses the safest legal action, `heuristic_guard` upgrades to a defensive action when available, and `last_valid_replayable` first replays the prior compatible decision before falling back to guard/safe behavior.
- `daemon/src/yomi_daemon/orchestrator.py` is now the async wrapper that converts provider timeouts, disconnects, malformed output, illegal output, and stale responses into structured fallback decisions instead of letting exceptions escape.
- Full Python gates passed from the repository root with `uv run --project daemon ruff format`, `uv run --project daemon ruff check`, `uv run --project daemon ty check daemon/src tests`, and `uv run --project daemon pytest tests`.
