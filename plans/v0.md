# YOMI Hustle LLM Arena - Implementation Plan v0

This plan translates `specs/unified_spec.md` into a commit-oriented execution plan for coding agents.

## Planning assumptions

- The repository currently contains only specifications, so early work units must establish project scaffolding, tooling, and test harnesses.
- The mod and daemon should evolve in parallel, but anything that depends on exact YOMI Hustle hook points must wait for game mapping output.
- Every work unit is intended to be a logical standalone commit with clear deliverables, tests, and dependency boundaries.
- The game remains the authority on legality; the daemon remains the authority on policy orchestration, logging, and tournament management.
- MVP transport is localhost WebSocket with the mod acting as client and the daemon acting as server.

## Global implementation rules

- Protocol changes must remain schema-first and versioned under `schemas/`.
- Match artifacts must be written under `runs/<timestamp>_<match_id>/` with enough metadata to audit every decision.
- All mod-applied decisions must be revalidated against live legal actions before they are injected.
- All Python dependency management and Python code execution should use `uv` (for example `uv sync`, `uv run pytest`, `uv run python ...`) rather than raw `pip`, `python`, or `pytest` invocations.
- Tests should be added as close as possible to the code they protect, with reusable fixtures under `tests/fixtures/`.
- Work units should prefer additive changes so multiple agents can work concurrently with minimal merge conflicts.

## Recommended execution order

- Start with `WU-000`, `WU-001`, and `WU-002`.
- Once `WU-000` and `WU-002` land, daemon work can split across `WU-003` to `WU-007`.
- Once `WU-000` and `WU-001` land, mod work can split across `WU-008` to `WU-010`.
- End-to-end runner, tournaments, and hardening should wait for both daemon and mod MVP paths to exist.

## WU-000 - Repository scaffold and toolchain baseline

- Blocked by: none
- Goal: create the repository layout, package boundaries, shared test setup, and CI skeleton described by the spec so later work units can land without restructuring churn.

### Scope

- Create the top-level directory structure from the spec: `docs/`, `schemas/`, `mod/`, `daemon/`, `prompts/`, `runs/`, `scripts/`, `tests/`, `.github/workflows/`.
- Add placeholder or initial files that define project ownership boundaries, including `README.md`, `daemon/pyproject.toml`, mod metadata stub, and `runs/.gitkeep`.
- Set up Python packaging and test tooling for the daemon, plus a CI workflow that runs schema checks and daemon tests.
- Add basic developer documentation describing how to run tests, where generated artifacts live, and which directories are authoritative for protocol, mod, and daemon code.

### Implementation details known ahead of time

- Keep Python source under `daemon/src/yomi_daemon/`.
- Reserve `mod/YomiLLMBridge/` for Godot files even if most files start as stubs.
- Python project setup should be `uv`-first so local development, CI, and coding agents all use the same commands for dependency management and execution.
- CI can initially limit itself to Python lint/test plus schema validation; game-driven mod integration can be added later.

### Acceptance criteria

- Repository tree matches the intended long-term shape closely enough that later work does not need large file moves.
- Daemon package installs and test commands execute successfully via `uv`.
- CI runs on pull requests and fails on broken Python tests or invalid schema files.
- README documents local setup for daemon development and references the unified spec.

### Tests to add

- A smoke test that imports `yomi_daemon` from the packaged source layout.
- A CI test fixture or script that validates every JSON schema file can be parsed.
- A CI or setup check that project commands are executed through `uv` entry points.

## WU-001 - Game internals mapping and bridge design docs

- Blocked by: `WU-000`
- Goal: de-risk the mod by documenting the concrete YOMI Hustle hook points, native decision pipeline, and legal action extraction strategy before implementing live bridge logic.

### Scope

- Produce `docs/game_internals.md` with the decompilation workflow, relevant classes/scripts, decision entry points, and move application path.
- Document how legal actions, move-specific payloads, DI, feint, reverse, and turn locking are represented in-game.
- Capture version assumptions for the supported game build and any fragility risks around mod loader overrides.
- Add notes on how to build deterministic observations from live game state and which fields may require normalization.

### Implementation details known ahead of time

- Use the spec's `scripts/decompile.sh` as the canonical entry point for repeatable inspection.
- The output should explicitly identify which future files own each concern: `TurnHook.gd`, `ObservationBuilder.gd`, `LegalActionBuilder.gd`, `ActionApplier.gd`.
- If exact symbols are unstable, document fallback discovery methods and smoke-test expectations rather than leaving ambiguity.

### Acceptance criteria

- The documentation is detailed enough that a coding agent can implement mod hook files without rereading the entire spec.
- Decision interception, legality enumeration, and action injection paths are each mapped to specific engine scripts or methods.
- Known game-version risks and validation strategies are called out.

### Tests to add

- A script-level smoke check that `scripts/decompile.sh` exists, runs, and writes output to a documented location.
- If possible, a lightweight regression fixture that records the discovered script names or hashes for the supported game build.

## WU-002 - Canonical protocol schemas and typed models

- Blocked by: `WU-000`
- Goal: define the shared wire contract once so daemon and mod work can proceed against a stable versioned interface.

### Scope

- Create versioned JSON schemas for envelope, hello/hello-ack, decision request, action decision, event, match ended, and config payloads.
- Implement typed Python protocol models and validation helpers in `daemon/src/yomi_daemon/protocol.py` and `validation.py`.
- Define canonical enums and field constraints for message type, protocol version, decision type, fallback reason, and bounded DI vectors.
- Document schema semantics in `docs/protocol.md`, including handshake compatibility rules and append-only versioning policy.

### Implementation details known ahead of time

- Envelope shape should be `{type, version, ts, payload}`.
- `DecisionRequest` must include `match_id`, `turn_id`, `player_id`, `deadline_ms`, `state_hash`, `legal_actions_hash`, `decision_type`, `observation`, and `legal_actions`.
- `ActionDecision` must include `action`, `data`, and `extra`, with optional debug metadata.
- Hashes should be treated as opaque strings in the schema layer; computation logic belongs elsewhere.

### Acceptance criteria

- Every required protocol message in the spec has a concrete schema file and a typed Python representation.
- Validation helpers reject malformed payloads with actionable error messages.
- Protocol documentation explains compatibility, negotiation, and the distinction between schema validation and mod-side legality validation.

### Tests to add

- Unit tests for valid and invalid examples of every core message type.
- Round-trip tests from Python typed model -> JSON payload -> schema validation.
- Negative tests for stale or mismatched version fields and malformed `extra.di` vectors.

## WU-003 - Daemon server, handshake, and match session skeleton

- Blocked by: `WU-000`, `WU-002`
- Goal: stand up the daemon's WebSocket server and session lifecycle so the mod has a stable peer for connection and handshake work.

### Scope

- Implement the daemon CLI entry point and WebSocket server in `cli.py` and `server.py`.
- Support localhost-only listening by default and reject unsupported protocol versions during handshake.
- Track active connections, session state, player-slot policy assignments, and current match metadata.
- Implement `Hello` -> `HelloAck` negotiation and basic lifecycle logging.

### Implementation details known ahead of time

- Bind to `127.0.0.1` by default; IPv6 loopback can be configurable later.
- Use one persistent connection per game process.
- Keep session state minimal in this work unit: accepted version, daemon version, assigned policies, and match config snapshot.
- Do not yet route to full policy logic; stub decisions are acceptable until later units land.

### Acceptance criteria

- The daemon starts from the CLI, accepts a local WebSocket connection, and completes a versioned handshake.
- Unsupported protocol or schema versions are rejected cleanly without crashing the server.
- Session state is stored in a way later routing and artifact code can extend without redesign.

### Tests to add

- Async integration tests for successful handshake, unsupported version rejection, and localhost binding defaults.
- Tests for player-slot policy mapping in `HelloAck`.

## WU-004 - Daemon config loading and match manifest foundations

- Blocked by: `WU-000`, `WU-002`
- Goal: define how the daemon is configured and how run metadata is pinned for reproducibility before richer orchestration is added.

### Scope

- Implement config schema loading for transport, timeout profiles, fallback mode, logging options, policy assignments, character selection, and tournament defaults.
- Add config parsing helpers and a normalized runtime config object.
- Create initial manifest generation utilities that pin schema/protocol version, daemon version, prompt version, policy IDs, and trace seed.
- Write `docs/architecture.md` or `docs/operations.md` sections describing config precedence and artifact layout.

### Implementation details known ahead of time

- Support named timeout profiles `strict_local` and `llm_tournament` with spec defaults.
- Keep provider credentials out of config files; reference environment variable names instead.
- Manifest data should be serializable before the first turn so incomplete matches are still auditable.

### Acceptance criteria

- Daemon config files validate and load into a typed runtime structure.
- The system can generate a manifest skeleton before a match starts.
- Invalid config values fail fast with human-readable errors.

### Tests to add

- Unit tests for default config expansion and environment-backed provider credential references.
- Negative tests for invalid timeout profiles, missing policy assignments, and malformed character selection modes.
- Tests that manifest generation includes required pinned metadata fields.

## WU-005 - Artifact writer and telemetry persistence

- Blocked by: `WU-000`, `WU-002`, `WU-004`
- Goal: guarantee that every match can emit auditable artifacts, even before full tournament support exists.

### Scope

- Implement `storage/writer.py` and `storage/replay_index.py` for creating per-match directories and writing `manifest.json`, `events.jsonl`, `decisions.jsonl`, `prompts.jsonl`, `metrics.json`, `result.json`, `replay_index.json`, and `stderr.log` placeholders.
- Define the daemon-side logging schema utilities in `logging/schemas.py`.
- Add append-safe writing APIs for request/response logs, telemetry events, and end-of-match summaries.
- Ensure the writer can finalize partial matches after error or disconnect scenarios.

### Implementation details known ahead of time

- Directory pattern should be `runs/<timestamp>_<match_id>/`.
- Request payloads and returned decisions must be stored exactly as sent/received, not just normalized summaries.
- File creation order should tolerate crashes and preserve partial state.

### Acceptance criteria

- Starting a match creates a complete artifact directory skeleton.
- Turn-level events, prompts, and decisions append as JSONL records without corrupting prior data.
- Match finalization writes `result.json` and updates metrics even after an error path.

### Tests to add

- Filesystem tests for directory creation, append order, and finalization on both success and failure.
- Snapshot-like tests for manifest and event record structure.
- Tests that exact request/response payloads are preserved in logged artifacts.

## WU-006 - Policy adapter base and deterministic baseline policies

- Blocked by: `WU-000`, `WU-002`, `WU-004`
- Goal: provide a stable adapter contract and enough baseline policies to exercise the system without external providers.

### Scope

- Implement `adapters/base.py` with the `PolicyAdapter` contract and shared metadata.
- Implement `adapters/baseline.py` with `baseline/random`, `baseline/block_always`, `baseline/greedy_damage`, and `baseline/scripted_safe`.
- Add deterministic seeding support tied to match `trace_seed` and turn context.
- Expose adapter construction from config and player-slot assignment.

### Implementation details known ahead of time

- Baselines operate purely on `DecisionRequest` data and should never rely on hidden engine state.
- `baseline/random` should be deterministic under the same seed and legal action ordering.
- `baseline/greedy_damage` should use tactical metadata when present, with documented fallback heuristics when absent.
- `baseline/scripted_safe` can initially be conservative and legality-preserving rather than strategically strong.

### Acceptance criteria

- All required baseline IDs from the spec are supported.
- Baselines return structured `ActionDecision` payloads that validate against schemas.
- Deterministic baselines produce identical outputs for the same seeded request sequence.

### Tests to add

- Unit tests per baseline policy covering empty metadata, tie-breaking, and unsupported extras.
- Determinism tests across repeated seeded runs.
- Contract tests that every adapter returns decisions matching the typed protocol model.

## WU-007 - Response parsing, daemon-side validation, and fallback engine

- Blocked by: `WU-002`, `WU-004`, `WU-006`
- Goal: centralize how the daemon parses model outputs, validates them against the request, and constructs explicit fallback decisions.

### Scope

- Implement `response_parser.py`, `fallback.py`, and validation utilities for matching `ActionDecision` values to the current request.
- Support provider-native structured output, JSON parsing, and constrained text extraction in descending preference order.
- Implement a single retry-with-correction path for invalid LLM output when configured.
- Encode layered fallback behavior: `safe_continue`, optional `heuristic_guard`, then `last_valid_replayable` if enabled and legal.

### Implementation details known ahead of time

- Daemon-side validation is advisory; mod-side legality checks remain final.
- Fallback outputs must carry explicit reason codes for timeout, disconnect, malformed output, illegal output, or stale response.
- `last_valid_replayable` should only reuse prior decisions when the same action remains legal under the current request.

### Acceptance criteria

- The daemon can turn invalid or late model output into a structured fallback decision without throwing uncaught exceptions.
- Retry behavior is bounded and config-driven.
- Validation detects mismatched `match_id`, `turn_id`, unsupported extras, and actions absent from the legal set.

### Tests to add

- Unit tests for structured parse success, JSON extraction, text extraction, correction retry, and fallback selection.
- Negative tests for stale IDs, illegal actions, malformed payloads, and out-of-bounds DI vectors.
- Property-style tests for `last_valid_replayable` legality checks.

## WU-008 - Prompt formatting and first provider-backed adapter

- Blocked by: `WU-002`, `WU-004`, `WU-007`
- Goal: support at least one real LLM-backed policy path while keeping prompting deterministic and auditable.

### Scope

- Implement `prompt.py` and prompt template files for `minimal`, `strategic`, `few_shot`, and `reasoning_enabled` variants.
- Implement one production-ready provider adapter first, plus reusable scaffolding for other providers under `adapters/`.
- Persist rendered prompts and provider metadata into match artifacts.
- Support provider-native structured output where available, with graceful fallback to parser-based extraction.

### Implementation details known ahead of time

- Choose one adapter category as the first full implementation; keep the interface generic enough for `anthropic/*`, `openai/*`, `google/*`, `ollama/*`, and `local/*` follow-up work.
- Prompt rendering should be deterministic for a given request, template version, and config.
- Provider credentials must be sourced from environment variables, not committed config.

### Acceptance criteria

- The daemon can format a request into a prompt, call one real provider adapter, parse the output, and return a schema-valid decision.
- Prompt text and relevant provider metadata are logged per turn.
- Missing credentials or provider errors degrade into structured fallbacks rather than crashing the match.

### Tests to add

- Unit tests for prompt rendering determinism and template selection.
- Mocked adapter tests for successful provider response, malformed provider response, timeout, and credential-missing paths.
- Artifact tests verifying prompt persistence and provider metadata capture.

## WU-009 - Mod skeleton, config, and WebSocket bridge client

- Blocked by: `WU-000`, `WU-001`, `WU-002`
- Goal: establish the Godot-side bridge foundation that can connect to the daemon, negotiate versions, and expose mod configuration surfaces.

### Scope

- Implement `_metadata`, `ModMain.gd`, `bridge/BridgeClient.gd`, and `ui/ModOptions.gd` at a skeleton-but-functional level.
- Load default config from `mod/YomiLLMBridge/config/default_config.json` and expose localhost host/port, timeout profile, and logging toggles.
- Connect to the daemon over WebSocket, send `Hello`, parse `HelloAck`, and maintain connection state.
- Add defensive handling for disconnects, malformed payloads, and unsupported version responses.

### Implementation details known ahead of time

- Default connection target must be `127.0.0.1` unless explicitly overridden.
- The mod should never execute arbitrary code from daemon payloads; all parsing must be data-only.
- Keep UI scope small in MVP: enough to inspect connection state and tweak core bridge settings.

### Acceptance criteria

- The mod loads with the bridge enabled, reads config, and completes a handshake against the daemon server.
- Connection failures and malformed handshake responses do not crash the game.
- Local-only defaults and version negotiation rules align with the protocol docs.

### Tests to add

- Script-level tests or harnesses for config loading and handshake message formatting.
- Mock-daemon integration tests for successful connection, unsupported version response, and disconnect handling.

## WU-010 - Mod turn hook, observation builder, and legal action enumeration

- Blocked by: `WU-001`, `WU-002`, `WU-009`
- Goal: capture actionable turn state deterministically and emit canonical `DecisionRequest` payloads from live gameplay.

### Scope

- Implement `TurnHook.gd`, `ObservationBuilder.gd`, and `LegalActionBuilder.gd` using the mappings documented in `WU-001`.
- Detect actionable decision points, assign stable `turn_id` values, and gather match/turn metadata.
- Build compact deterministic observations for fighters, objects, stage data, and recent history.
- Enumerate legal actions with canonical IDs, payload specs, supported extras, and any tactical metadata available in-game.
- Compute and attach `state_hash` and `legal_actions_hash`.

### Implementation details known ahead of time

- Observation and legal action ordering must be stable so determinism tests remain meaningful.
- Hash inputs should be canonicalized serialized payloads, not engine object identity.
- Tactical metadata should be included when cheap and available, but legality data takes precedence over enrichment.

### Acceptance criteria

- At each actionable turn, the mod can emit a schema-valid `DecisionRequest` built from live game state.
- Repeated identical states produce identical observation ordering and hash values.
- Legal actions include enough structure to support action payloads, DI, feint, and reverse when available.

### Tests to add

- Unit or fixture-driven tests for observation normalization and legal-action canonicalization.
- Determinism tests for stable hashes across identical synthetic states.
- Integration tests with a mock daemon verifying emitted `DecisionRequest` payload shape.

## WU-011 - Mod decision validation, action application, fallback behavior, and telemetry

- Blocked by: `WU-002`, `WU-009`, `WU-010`, `WU-007`
- Goal: make the mod safe in live gameplay by revalidating every response, applying only legal actions, and degrading gracefully when anything goes wrong.

### Scope

- Implement `DecisionValidator.gd`, `ActionApplier.gd`, `FallbackHandler.gd`, and `Telemetry.gd`.
- Validate incoming decisions for version, `match_id`, `turn_id`, action legality, supported extras, and stale-turn freshness before applying.
- Inject legal decisions into the game's native decision pipeline and lock them in through the same path as normal play.
- Apply layered fallback behavior on timeout, disconnect, malformed payload, or illegal response.
- Emit telemetry events including `TurnRequested`, `DecisionReceived`, `DecisionApplied`, `DecisionFallback`, `MatchEnded`, and `Error`.

### Implementation details known ahead of time

- Mod-side legality remains authoritative even if the daemon claimed validation success.
- Fallback reason codes should align with daemon logging so artifacts can be correlated across both sides.
- Telemetry should include latency, hashes, and legality verdicts where available.

### Acceptance criteria

- Illegal, stale, or malformed decisions are never applied to live game state.
- Timeout and disconnect scenarios resolve through safe fallbacks instead of deadlocking the match.
- Telemetry events are emitted consistently for both normal and failure paths.

### Tests to add

- Validation tests for stale `turn_id`, wrong `match_id`, unsupported extras, and illegal actions.
- Integration tests for timeout, disconnect, malformed payload, and successful action application.
- Reliability tests asserting bad daemon payloads never crash the mod.

## WU-012 - End-to-end single-match orchestration and local run scripts

- Blocked by: `WU-003`, `WU-005`, `WU-006`, `WU-008`, `WU-011`
- Goal: make a full autonomous head-to-head match runnable from the repository with auditable outputs.

### Scope

- Implement match orchestration in `orchestrator.py`, `match.py`, and helper scripts such as `scripts/run_local_match.sh`.
- Wire together session startup, policy assignment, turn routing, artifact writing, and match finalization.
- Support at least baseline-vs-baseline and baseline-vs-provider runs.
- Capture end-of-match summary data including winner, end reason, turns, duration, average latency, fallback counts, and token/cost totals where available.

### Implementation details known ahead of time

- The first runnable path can assume a single game instance and one daemon process.
- Match orchestration should tolerate partial failures and still produce result artifacts.
- Replay pointer support can begin as an index placeholder if the game-side replay extraction is deferred.

### Acceptance criteria

- A documented command can run one autonomous match end to end and create the expected artifact directory.
- Match results include complete per-turn request and decision records plus a final result summary.
- Timeout, disconnect, and invalid-output scenarios still terminate cleanly with logged fallback information.

### Tests to add

- End-to-end integration tests with a mock or fixture-backed mod request loop.
- Artifact completeness tests covering all required files for a finished match.
- Smoke tests for baseline-vs-baseline orchestration.

## WU-013 - Tournament scheduler, Elo ratings, and report generation

- Blocked by: `WU-005`, `WU-006`, `WU-012`
- Goal: scale from one match to reproducible comparison runs with ratings and aggregated reporting.

### Scope

- Implement `tournament/scheduler.py`, `runner.py`, `ratings.py`, and `reporter.py`.
- Support single match, side-swapped pair, round robin, and double round robin scheduling.
- Add Elo updates, raw win-rate aggregation, match counts, latency summaries, fallback summaries, and matchup tables.
- Add a helper script such as `scripts/run_round_robin.sh`.

### Implementation details known ahead of time

- MVP tournament defaults should prefer mirror matches, side swap enabled, fixed stage, and configurable games per pair.
- Concurrency should be conservative at first to respect provider rate limits.
- Ratings should remain reproducible from persisted match results rather than in-memory-only state.

### Acceptance criteria

- The system can execute a multi-match tournament and emit leaderboard plus matchup summary outputs.
- Elo ratings, win rates, and match counts are derived from recorded results and can be recomputed.
- Side-swapped scheduling works and is visible in reports.

### Tests to add

- Unit tests for scheduler pairings across supported formats.
- Ratings tests with known Elo examples.
- Reporter tests for summary tables and matchup aggregation.

## WU-014 - Hardening, determinism, and performance verification

- Blocked by: `WU-010`, `WU-011`, `WU-012`, `WU-013`
- Goal: close the MVP quality gap by proving the system meets reliability, determinism, and latency expectations from the spec.

### Scope

- Add determinism and reliability suites across baseline runs, artifact manifests, hash consistency, malformed payload handling, and disconnect behavior.
- Add performance instrumentation and benchmark scripts for `strict_local` and `llm_tournament` profiles.
- Tighten docs in `docs/operations.md` and `README.md` with troubleshooting, known limitations, and reproducibility guidance.
- Resolve any remaining mismatches between implementation and the unified spec discovered during end-to-end validation.

### Implementation details known ahead of time

- Determinism verification should focus on seeded baseline-vs-baseline runs because provider-backed runs are inherently partially non-deterministic.
- Performance reporting should surface p95 round-trip latency and fallback rate by profile.
- Any non-deterministic fields in artifacts should be clearly separated from reproducible metadata.

### Acceptance criteria

- Seeded baseline runs reproduce identical decision logs and compatible manifests.
- Malformed daemon payloads, invalid actions, and disconnects are all handled without mod crashes or illegal action application.
- Benchmarks demonstrate conformance or clearly report gaps against the spec budgets.

### Tests to add

- Determinism suites comparing repeated seeded match outputs.
- Reliability integration tests for timeout, disconnect, malformed payload, and hash mismatch handling.
- Benchmark or regression tests for latency and fallback-rate reporting.
